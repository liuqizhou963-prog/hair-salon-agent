# 技术选型与面试问答

## 为什么用 FastAPI

FastAPI 的类型标注和 Pydantic 校验适合快速构建清晰的 API 合同。接口输入错误时可以直接返回 422，Swagger 文档也能用于面试演示和调试。

## 为什么用 SQLAlchemy

SQLAlchemy 把数据库表映射成 Python 模型，同时保留事务、查询和关系能力。Agent 不直接拼 SQL，而是调用业务服务或只读工具，权限边界更容易集中管理。

## 为什么正式环境用 PostgreSQL，测试用 SQLite

PostgreSQL 更适合并发、事务和正式部署；SQLite 内存库启动快、隔离简单，适合 pytest 每个测试重建数据。代码通过 `DATABASE_URL` 切换，Alembic 负责结构版本。

## LangChain 和 LangGraph 的关系

LangChain 是模型、消息、工具和结构化输出的基础组件集合；LangGraph 用节点和边把这些组件编排成可控流程。本项目的员工查询 Graph 是“意图识别 -> 工具查询 -> 结果整理”，预约调整 Graph 还增加了人工确认和事务执行分支。

## 为什么不用大模型直接操作数据库

大模型可能误解问题、编造 ID 或重复执行写操作。系统只把受控工具暴露给 Agent，工具内部仍然做权限和数据校验；关键写操作还必须经过人工确认和数据库事务。

## RAG 和数据库查询有什么区别

RAG 适合护理规则、服务说明等相对稳定的文本知识；预约、余额、积分和通知是实时业务数据，必须从数据库查询。把实时数据塞进知识库会导致过期和错误。

## 为什么保留 BM25 兜底

Chroma 需要 embedding 服务或 Ollama，面试现场不一定有网络或模型。BM25 使用本地分词和索引，不需要 Key，能保证没有外部模型时仍然有可解释的离线检索结果。

## 为什么暂时不接 MCP

当前工具都在同一个后端进程内，LangGraph 直接调用 Python 工具即可。MCP 更适合多个 Agent 或外部系统跨进程共享工具；现在引入 MCP 会增加协议和部署复杂度，却不会改善 v1 的业务闭环。

## 如何保证预约调整的数据一致性

提议阶段只读数据库并保存任务状态；确认阶段重新检查预约状态和新时间槽是否被占用，然后在同一个 Session 中释放旧槽、占用新槽、修改预约、写审计和写通知。任何一步失败就回滚。

## 如何排查 Agent 答案不对

先看返回的 `actions` 判断调用了哪个工具，再看 `sources` 判断数据来源；然后单独测试工具，再测试 Graph 路由，最后才检查模型 Prompt。这样可以把“意图判断错误”和“业务查询错误”分开定位。
