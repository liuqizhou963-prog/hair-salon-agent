# Agent 评测方案

这份方案用于回答一个面试官常问的问题：Agent 看起来能回答，不等于它真的可靠。项目需要用固定问题、固定预期和可重复的结果来评估 Agent。

## 1. 评测目标

- 判断问题是否被路由到正确的意图和工具。
- 判断工具是否只读取允许的数据，没有越权写入预约、钱包或通知。
- 判断 RAG 是否召回了相关知识，而不是只返回一段看似通顺的文字。
- 判断员工确认前，预约调整是否不会真正落库。
- 判断失败时是否能留下任务状态和 trace，便于定位问题。

## 2. 固定评测集

### 员工只读 Agent

| 场景 | 示例问题 | 关键预期 |
| --- | --- | --- |
| 日程查询 | 今天有哪些预约？ | 意图为 `schedule`，调用排班查询工具 |
| 客户记录 | 13900000001 最近有没有预约？ | 意图为 `customer`，只查询客户和预约 |
| 会员信息 | 这个客户的余额和积分是多少？ | 意图为 `membership`，调用会员和钱包查询 |
| 留存提醒 | 现在有哪些留存提醒？ | 意图为 `retention`，读取待跟进提醒 |
| 染发知识 | 漂到橙黄色应该怎么校色？ | 意图为 `knowledge`，调用 RAG 并返回相关来源 |
| 不明确问题 | 你好 | 意图为 `help`，不调用写入工具 |

### 需要人工确认的 Agent

| 场景 | 操作 | 关键预期 |
| --- | --- | --- |
| 生成方案 | 提交新的时间槽 | 只生成 proposal，预约仍保持原时间 |
| 员工拒绝 | 确认接口传 `confirmed=false` | 任务结束，但数据库不发生预约调整 |
| 员工确认 | 确认接口传 `confirmed=true` | 在事务中锁定预约和目标时间槽后修改 |
| 目标空档被抢 | 两次确认同一个空档 | 后到请求失败，不能产生重复预约 |

## 3. 评测指标

- **路由准确率**：固定问题进入预期意图的比例，目标为 100%。
- **工具准确率**：是否调用正确的受控工具，是否没有调用不允许的写工具。
- **RAG 命中率**：固定问题的第一条来源是否为预期知识，当前用标题和来源断言验证。
- **安全通过率**：未登录、客户访问员工接口、未确认写操作等场景必须被拒绝。
- **可追踪率**：成功和失败的 Agent 任务都能通过任务记录或日志定位到请求、节点和结果。

## 4. 评测方式

1. 使用 `pytest` 执行固定评测集，保证每次改 Prompt、路由或知识库后都能回归。
2. 对线上真实问题做脱敏抽样，新增到评测集，不直接把客户隐私写进代码。
3. 发现失败样本时，先看 `trace` 判断是路由错、工具错、检索错还是格式化错，再决定改哪一层。
4. 任何会触发预约、钱包、通知的能力，都必须单独测试人工确认和事务边界。

## 5. 面试话术

> 我没有只用“回答得像不像人”来评估 Agent，而是把评测拆成路由、工具、RAG、安全和可追踪五类指标。每类都有固定样例和自动化断言。这样改了模型、提示词或知识库之后，可以知道到底是哪一层退化，也能证明 Agent 没有越权执行写操作。
