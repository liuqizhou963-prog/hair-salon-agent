# 恒艺美发架构与流程

## 1. 总体架构

```mermaid
flowchart LR
  C[客户 H5 / 小程序] --> A[FastAPI API]
  S[员工工作台] --> A
  A --> Auth[JWT 与角色权限]
  A --> Biz[业务服务层]
  Biz --> DB[(SQLite / PostgreSQL)]
  A --> G1[员工查询 Graph]
  A --> G2[预约调整 Graph]
  A --> G3[留存运营 Graph]
  G1 --> T1[只读数据库工具]
  G1 --> RAG[RAG 检索]
  G2 --> Human[人工确认节点]
  Human --> Tx[预约事务执行器]
  G3 --> Segment[客户分层节点]
  Tx --> DB
  Tx --> N[站内通知与审计]
  Segment --> Task[AgentTaskState 执行记录]
```

## 2. 员工只读查询

```mermaid
flowchart TD
  Q[员工问题] --> I[意图识别]
  I -->|预约/日程| D1[查询员工日程]
  I -->|客户/历史| D2[查询客户与预约]
  I -->|会员/余额/积分| D3[查询会员与钱包]
  I -->|留存/运营| D4[查询待跟进提醒]
  I -->|护理知识| R[检索 RAG]
  D1 --> F[整理回答]
  D2 --> F
  D3 --> F
  D4 --> F
  R --> F
  F --> O[reply + actions + sources]
```

数据库查询和 RAG 查询必须分开。预约、余额、积分是实时数据，必须查数据库；护理规则和知识是文档数据，使用 RAG 检索。

## 3. 预约调整与人工确认

```mermaid
sequenceDiagram
  participant Staff as 员工
  participant Graph as Appointment Graph
  participant Task as AgentTaskState
  participant DB as Database
  participant Client as 客户通知

  Staff->>Graph: 提交预约 ID 与新时间槽
  Graph->>DB: 读取预约和时间槽
  Graph->>Task: 保存 awaiting_confirmation
  Graph-->>Staff: 返回调整方案，不写业务数据
  Staff->>Task: confirmed=true
  Task->>DB: 重新校验时间槽占用状态
  DB->>DB: 释放旧槽、占用新槽、更新预约
  DB->>DB: 写审计记录
  DB->>Client: 写站内通知
  Task-->>Staff: completed
```

确认接口会重新校验，而不是相信提议阶段的旧结果。这样可以处理员工等待确认期间时间槽被其他人占用的情况。

## 4. RAG 降级策略

```text
护理问题
  -> Chroma 向量检索（启用并且 embedding 可用）
  -> 失败或未启用
  -> 本地 jieba + BM25 检索
  -> 仍没有结果
  -> 返回明确的“未检索到”提示
```

MCP 当前没有接入。项目内部工具由 LangGraph 直接调用已经满足 v1 目标；未来如果多个 Agent 需要跨进程共享数据库、文件或第三方系统工具，再增加 MCP Server/Client。
