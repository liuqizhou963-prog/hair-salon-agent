# 留存 Agent 开发流程

> 配套文档：`docs/留存提醒完整优化方案.md`（方案评审稿）
> 本文只讲「怎么落地」：按阶段拆任务、对齐现有代码、每步给验收口径。
> 现状基线：留存逻辑已存在于 `backend/database/retention.py`（纯规则引擎）+ `backend/agents/retention_graph.py`（分层 Graph），本次是在其上重构，不是从零写。

## 零、开工前必须先拍板的 4 个决策

这几项不定，后面全是返工。建议开发前找业务/面试对齐，定不了就按「建议值」写成明确常量并在待确认项登记。

| 决策 | 现状 | 方案主张 | 本流程建议 |
|---|---|---|---|
| 流失边界 | `days_since >= cycle × 2.5`（跟个人周期） | 固定 `>= 150 天` | **保留个人周期兜底**：`max(150, 3 × cycle)`，避免高频客户被判太晚、长周期客户复购窗口过长 |
| 冷却期存放位置 | 无（只有 14 天同类去重） | task 与 contact 都提到 next_contact_at | **权威来源定在客户级**：查询时取该客户 `max(sent_at + cooldown)`，任务上的字段只作快照 |
| 扫描幂等键 | 无唯一约束 | 一天一任务 | 唯一约束 `(customer_id, business_date)`，**不带 type** |
| v1 话术来源 | 纯模板（含违规文案） | Agent 生成，模型不可用时降级 | **安全模板为主路径**，LLM 为可选增强（当前 env 无 `LLM_API_KEY`） |

> 决策记录后，同步更新方案 §二十「默认假设与待确认项」。

## 第一阶段：统一规则引擎（不碰页面）

目标：先把「谁今天该出现」算对、算得可测。改 `backend/database/retention.py` 与 `backend/agents/retention_graph.py`。

### 任务清单
1. **修流失边界**：按上表决策改 `evaluate_customer`，同步 `retention_graph.py:40-66` 里 `analysis_basis.rules` 的文字描述（现在写的是 2.5 倍，会和代码不一致）。
2. **落地统一拦截优先级**：新增前置检查函数，顺序 `退订 > 永久忽略 > 限时忽略 > 人工跟进 > 全局冷却 > 业务规则`。命中任一即不进候选。
3. **候选合并**：同一客户多规则命中时，按 `生日 > 有余额流失 > 普通流失 > 复购` 选一个主类型，其余转为辅助标签（`trigger_reasons`）。
4. **删除 balance_customer 独立候选**：`retention_graph.py:79-90` 现在把余额客户 append 成独立条目，改为合并进主任务的标签。
5. **删除违规文案**：`retention.py:136` 的"给您备了一份生日护理"必须去掉，改为不含具体权益的关怀模板。
6. **复购防刷屏**：加"复购连续提醒 ≤ N 次后降级静默"逻辑。
7. **补纯规则单测**（不需要 LLM key，可立即跑）。

### 验收（对应方案 §17.1-17.4，用 pytest）
- 生日：还剩 6 天不出现、5 天内出现、今天出现；同年只成功发一次；生日+流失同命中只生成 1 条。
- 复购：未达阈值不出现、达标且未到流失线出现、历史不足走服务默认周期、重新到店后旧任务关闭。
- 流失：边界值 149/150 天分界正确；无余额走长冷却、有余额走短冷却+余额标签；余额客户不生成第 4 条任务。
- 全局：退订永不出现、30/90/永久忽略分别生效、发送失败不进冷却、同客户同天仅 1 条、重复扫描不产生重复数据。

```bash
PYTHONUTF8=1 python -m pytest tests/test_retention_agent.py -q
```

> 出口标准：新老单测全绿；本阶段结束前不改前端。

## 第二阶段：数据与接口补齐

目标：让系统能回答"谁审核、发了什么、是否失败、何时可再联系"。现有 `ReminderLog` 三态不够（`models.py:37-41`）。

### 任务清单
1. **扩展任务状态枚举** `ReminderStatus`：补 `sending / sent / send_failed / replied / manual_followup / cooling / closed`（对应方案 §10）。
2. **新增三张表 / 或在现表基础上拆**（对应方案 §13）：
   - `retention_tasks`：主类型 + 辅助标签 + 触发证据快照 + 优先级 + 状态 + 建议消息/优惠券 + `next_contact_at`；唯一约束 `(customer_id, business_date)`。
   - `retention_contacts`：审核人/发送人/渠道 + **实际发送内容**（非草稿）+ 实际优惠券 + 发送开始/成功/失败时间 + 渠道消息号 + 失败原因 + 客户回复。
   - `retention_suppressions`：退订/永久忽略/限时忽略/人工跟进 + 起止时间 + 原因 + 操作人 + 解除信息。
3. **写 alembic 迁移**（参照 `migrations/versions/` 现有风格，如 `0005_add_member_expiry.py`）。
4. **实现状态变更接口**（方案 §14），全部要求：登录身份 + 参数校验 + 当前状态校验 + 幂等/防重 + 审计日志（复用 `FinanceService.create_audit`）：
   - `POST /api/retention/scan`（幂等）
   - `POST /tasks/{id}/send`、`/retry`、`/manual-followup`、`/ignore`、`/reply`、`/close`
5. **发送闭环用可替换发送器**（方案 §15）：先 `MockMessageSender`，接口留给 `WeChatMessageSender`。冷却期只在**发送成功**时写入。

### 验收
- 迁移可正向+回滚。
- 重复调 `/scan` 不产生重复任务（幂等键生效）。
- 并发点两次 `/send` 只有一次真正发送（防重）。
- 发送失败任务留在列表可重试、且未写冷却。
- 每个状态变更都有 AuditLog。

## 第三阶段：收紧 Agent 边界

目标：Agent 只处理规则筛完的客户，结构化输入输出，优惠券必须来自有效列表。改 `backend/agents/retention_graph.py`、`backend/agents/tools.py`。

### 任务清单
1. **结构化输入/输出**：按方案 §11.3/§11.4 定 schema。输入含 customer、primary_type、trigger_reasons、cycle_days、balance、contact_history、available_coupons；输出含 strategy、message、coupon_id、coupon_reason、risk_flags。
2. **优惠券后端二次校验**：Agent 输出的 `coupon_id` 必须经后端确认真实、有效、有库存、适用当前客户才允许发送（方案 §11.4 末）。
   - ⚠️ 前置依赖：当前无 Coupon/Product 表（见项目记忆第 3 步）。若本阶段要真做优惠券，需先建券表；否则 v1 先让 `coupon_id=null`、只做话术。
3. **安全降级**：无 `LLM_API_KEY` 或模型超时/异常时，保留规则任务，回落到安全模板，绝不阻塞工作台（沿用 `langchain_agent.py` 现有降级模式）。
4. **Agent 不可越权**：不得绕过退订/忽略/冷却、不得改 150 天与冷却天数、不得虚构生日/余额/优惠。用测试固化这些边界。

### 验收
- 无 key 环境：Agent 路径静默降级，任务照常生成、话术为安全模板。
- 喂入不存在/过期券号，后端拦截、不发送。
- 结构化输出 schema 校验通过。

```bash
PYTHONUTF8=1 python -m pytest tests/ -q   # 期望全绿，含降级路径
```

## 第四阶段：重构留存工作台（前端）

目标：从"三个名单"变成"今日待办表"。对应方案 §12。前端在 `frontend/`。

### 任务清单
1. 两个主视图：**今日待处理**（默认，只显示今天可处理）+ **联系记录**（历史/冷却/忽略/失败）。
2. 顶部统计条 + 分段筛选（全部/生日/复购/流失）+ 辅助筛选（姓名手机搜索、负责人、余额客户、优先级、发送失败）。
3. 主表格字段按 §12.3；点击行开**详情抽屉**（不跳离列表），含可编辑消息框、有效优惠券选择器、Agent 理由与风险提示、发送/转人工/忽略按钮。
4. **忽略表单**（§12.5）：30/90/永久 + 原因；选"客户明确拒绝"时提示是否记为永久退订，退订走二次确认 + 审计。
5. 规则说明收进侧栏按钮，不占首屏。

### 验收
- 默认只出现今天可处理客户。
- 每个客户能点开看到"为什么出现在这"（触发证据）。
- 移动端、键盘操作、错误提示可用。

## 阶段依赖与节奏建议

```
第一阶段(规则,可独立测) ──► 第二阶段(数据+接口) ──► 第三阶段(Agent边界)
                                    └──────────────► 第四阶段(前端) ──► 第五阶段(联调)
```

- 第一阶段最该先做，纯规则、无 key 可测、风险最低、收益最大（顺手清掉现有违规文案和 balance 重复 bug）。
- 优惠券真做与否是第三阶段的岔路口，取决于是否先建券表；v1 可先 `coupon_id=null` 只做话术。
- 每阶段结束都以 `pytest tests/ -q` 全绿为出口，改动 Agent 后必须验证无 key 降级路径。
