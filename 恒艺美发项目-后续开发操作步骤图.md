# 恒艺美发项目后续开发操作步骤图

## 项目目标

把已经完成的 H5 和小程序，从“前端展示 + 演示数据”升级成一个完整可面试演示的真实业务系统：后端负责账号、预约、会员、钱包、通知、审计和运营数据；LangGraph Agent 负责员工辅助查询、预约调整确认、客户流失运营；数据库以 PostgreSQL 为正式环境，SQLite 用于测试隔离。最终目标不是只把功能做出来，而是让你能在面试中讲清楚每个模块为什么这样设计。

## 总执行规则

每一步都按这个顺序走：

1. 先解释这一阶段要做什么、为什么做、和其他模块有什么关系。
2. 你确认后再进入实现。
3. 复杂模块先画逻辑图，尤其是 Agent、RAG、API 路由和数据库事务。
4. 先补或确认测试思路，再改代码。
5. 完成后运行验证。
6. 最后总结这一阶段的面试讲法。
7. 你验收通过后，再进入下一步。

## 最终演示流程

1. 客户在 H5 或小程序注册/登录。
2. 客户查看服务、发型师和可预约时间。
3. 客户创建预约，后端写入真实数据库。
4. 员工在管理 Web 查看当天预约和客户信息。
5. 员工向 Agent 询问客户、预约或门店规则。
6. Agent 给出预约调整建议，但不直接写库。
7. 员工确认后，系统修改预约、写审计记录、发客户站内通知。
8. 客户端刷新后看到预约变化和通知。
9. 留存 Agent 识别需要运营的客户，生成可解释建议。
10. 面试时展示架构图、数据流、Agent 安全边界和测试结果。

## 已确认的范围边界

| 模块 | v1 做 | v1 暂不做 |
| --- | --- | --- |
| 登录 | 手机号 + 密码 | 微信登录、短信验证码 |
| 支付 | 可审计的演示充值/退款流水 | 真实微信支付 |
| 通知 | 站内通知、刷新或轮询 | WebSocket、微信订阅消息、短信 |
| 数据库 | PostgreSQL 正式环境，SQLite 测试 | Redis、Celery、复杂分库 |
| Agent | LangGraph 工具调用、人工确认、审计 | 让 AI 绕过确认直接写关键业务 |
| 权限 | 客户和员工/管理员基础区分 | 完整复杂 RBAC |
| RAG | 复用已有规则/知识检索能力 | 引入 Chroma 等新向量库 |
