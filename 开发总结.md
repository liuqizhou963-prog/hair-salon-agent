# 恒艺美发项目开发总结

这份文件是整个项目的“学习记录”。每完成一个步骤，就在这里追加一段，保证我能看懂项目为什么这样改，而不是只看到最后的代码。

## 当前项目基线

- 后端使用 FastAPI、SQLAlchemy、SQLite/PostgreSQL 配置、规则型 ChatAgent、LangChain 适配器和 RAG 检索。
- 后端已有预约、发型师空档、会员积分、生日营销和留存提醒接口。
- H5 和小程序仍有演示数据，后续需要逐步改成真实 API。
- 当前测试基线为 `8 passed`。
- 已发现测试会操作根目录 `hair_salon.db`，因此第一阶段先解决测试数据库隔离。

## 开发步骤记录

### 第 1 步：数据库与测试基础（已完成）

目标：让自动化测试使用独立数据库，并准备 Alembic 迁移基础，避免测试污染演示数据，也为后续登录、钱包和通知等新表做准备。

#### 做了什么

1. 在 `tests/conftest.py` 中让 pytest 在导入后端前固定使用 `sqlite://`，也就是共享内存数据库。
2. 在 `backend/database/connection.py` 中为内存 SQLite 配置 `StaticPool`，保证不同数据库会话看到同一份测试数据。
3. 每个测试开始前先删表、建表、填充发型师和时间槽示例数据，所以测试之间不会互相污染。
4. 新增 `tests/test_database_isolation.py`，明确检查测试不能连接项目根目录的 `hair_salon.db`。
5. 增加 Alembic 依赖、`alembic.ini`、迁移环境和 `0001_initial_schema` 初始迁移。
6. 在 `AGENTS.md` 增加规则：每完成一步都要在本文件里用小白能看懂的话记录原因、关联、验证和面试讲法。

#### 为什么这么做

以前 pytest 会直接操作根目录的演示数据库，测试可能新增预约、取消预约甚至重建表，导致下一次打开系统时数据变样。现在测试使用内存数据库，程序正式运行时仍然可以通过 `DATABASE_URL` 使用 PostgreSQL 或开发 SQLite。

Alembic 的作用是给数据库结构建立“版本号”。以后增加密码、钱包、通知等表时，可以执行迁移逐步升级数据库，不需要手工删库重建。

#### 和其他模块的关系

数据库连接是所有 API、Agent、会员服务和预约服务的底层依赖。先把测试和迁移基础稳定下来，后续登录、权限、钱包和通知的改动才有可靠的回归验证环境。

#### 验证结果

- `pytest tests/test_database_isolation.py::test_tests_use_a_separate_sqlite_database -q`：通过。
- `pytest -q`：`9 passed`。
- `DATABASE_URL=sqlite:///D:/面试项目/.alembic-verify.db alembic upgrade head`：通过。
- Alembic `downgrade base` 后再次 `upgrade head`：通过。
- 仍有 9 条警告：1 条 TestClient 依赖弃用警告，8 条循环外键删除表警告；它们不影响本步功能，但后续可以单独整理。

#### 面试讲法

> 我先把运行数据库和测试数据库隔离。正式环境可以连接 PostgreSQL，测试使用共享内存 SQLite，并且每个测试前重建数据。数据库结构通过 Alembic 版本化，这样后续新增登录、钱包和通知表时，不需要手工删库，测试也不会污染演示数据。

#### 剩余问题和下一步

- 生产环境的 PostgreSQL 连接参数还需要根据本机账号写入未提交的 `.env`，不能写入模板或代码。
- 现有业务模型还没有密码、钱包、退款、通知和审计字段。
- 下一步进入客户/员工登录和接口鉴权设计；预约归属校验会作为明确的安全验收项。

### 第 2 步：客户/员工登录与预约鉴权（已完成）

#### 做了什么

1. `User` 增加 `password_hash` 和 `is_active`，数据库迁移为 `0002_add_user_auth_fields`。
2. 增加 Argon2 密码哈希和 JWT 工具，密码不会以明文保存。
3. 增加登录接口：
   - `POST /api/auth/register`：客户注册。
   - `POST /api/auth/login`：客户、发型师、管理员统一登录。
   - `GET /api/auth/me`：查看当前登录用户。
4. 增加角色依赖：客户、员工、管理员分别有不同访问范围。
5. 预约创建、查询、取消都从登录 token 获取客户身份，不再相信请求体中的手机号。
6. 客户聊天和 LangChain 聊天接口也要求登录；员工身份会自动使用员工助手角色，不能由客户请求体伪造。
7. 客户列表、生日营销、留存工作台和员工日程增加员工权限；数据库初始化增加管理员权限检查。
8. 环境模板增加 JWT 密钥、员工演示密码和管理员演示账号配置。
9. 增加重复注册、错误密码、未登录访问、客户越权和迁移兼容测试。

#### 为什么这么做

以前接口只要传一个手机号就能访问数据，知道预约 ID 甚至可能取消别人的预约。现在 token 代表真实登录用户，后端从 token 查出用户和角色，再决定能不能执行操作。

密码使用 Argon2 哈希，即使数据库泄露，也不会直接暴露用户密码。JWT 适合 H5 和小程序，因为两个客户端都可以在请求头中携带同一种登录凭证。

#### 和其他模块的关系

登录身份是钱包、退款、通知、审计和前端真实 API 的共同基础。后续所有客户数据都应该通过当前用户 ID 查询，员工工作台则通过角色依赖保护。

#### 验证结果

- `pytest -q`：`15 passed`。
- `pytest tests/test_migrations.py -q`：通过，SQLite 空库可以升级到最新迁移。
- `python -m compileall -q backend migrations`：通过。
- 已验证客户不能取消他人预约，不能访问员工客户列表。

#### 面试讲法

> 我使用 JWT 建立当前用户身份，密码只保存 Argon2 哈希。API 通过统一依赖解析 token 和角色，预约查询和取消都用当前用户 ID 做归属校验，避免只相信前端传来的手机号。客户、发型师和管理员共用登录入口，但通过角色依赖访问不同业务范围。

#### 剩余问题和下一步

- H5 还没有登录页面和 token 请求封装，仍有 `demo-*` 发型师和时间槽数据。
- 小程序还没有真实请求封装，仍主要依赖 `globalData`。
- 员工演示账号需要在本机 `.env` 填写密码后，由初始化脚本创建。
- 下一步把 H5 和小程序的发型师、时间槽、预约和登录接到真实 API。

### 第 3 步：H5 与小程序接入真实预约 API（已完成）

#### 做了什么

1. H5 新增登录/注册页，使用后端 JWT；token 保存在浏览器 `localStorage`，请求自动带 `Authorization`。
2. H5 启动时调用 `/api/auth/me` 和 `/api/members` 恢复当前用户。
3. H5 发型师列表改为调用 `/api/stylists`，排档改为调用 `/api/stylists/{stylist_id}/slots`。
4. H5 预约列表改为调用 `/api/appointments`，预约按钮只调用真实 `POST /api/appointments`，删除了 `demo-*` 分支。
5. 修正 H5 发型师图片路径为后端静态目录 `/static/assets/...`，并通过浏览器验证资源不再 404。
6. 小程序新增登录页和注册/登录流程，token 保存到微信本地存储。
7. 小程序 `app.js` 增加统一 `wx.request` 封装：自动拼接 API 地址、附带 token、处理 401 和错误信息。
8. 小程序预约页、老师排档页、我的预约页和历史页改为读取真实后端数据，不再依赖本地发型师和时间槽数组。
9. 正式演示 SQLite 旧库先执行 `alembic stamp 0001_initial_schema`，再升级到 `0002`，没有删除原有数据。

#### 为什么这么做

之前两个客户端各自维护一套假数据，前端显示的老师和时间可能根本不存在于数据库。现在客户端只负责展示，发型师、排档、预约状态都来自同一套后端接口，客户预约后刷新页面仍能看到真实记录。

#### 和其他模块的关系

客户端真实 API 依赖第 2 步的 JWT 身份；预约接口又依赖后端的归属校验。后续钱包、通知和员工管理端也会复用同一个请求封装和登录 token。

#### 验证结果

- Playwright 浏览器流程：未登录显示登录页；注册并登录后显示当前用户；预约页加载真实张三等发型师；排档页加载数据库时间槽；点击时间槽后“我的预约”出现真实预约。
- H5 控制台资源 404 已修复。
- `pytest -q`：`15 passed`。
- 小程序全部 JS 通过 `node --check`。
- 小程序全部 JSON 使用 Node `JSON.parse` 验证通过。
- `rg "demo-" frontend`：无演示 ID。

#### 面试讲法

> H5 和小程序没有各自维护业务数据，而是共享后端 API。登录后客户端保存 JWT，请求自动带身份；发型师、时间槽和预约都从数据库读取，所以预约结果在不同客户端刷新后仍然一致。前端只负责页面状态，后端负责业务事实和权限校验。

#### 剩余问题和下一步

- 小程序的 `apiBase` 需要按微信开发者工具或真机网络环境改成后端局域网地址。
- 充值、退款、积分流水和通知页面还在使用演示状态，下一步需要先补钱包流水、退款申请和站内通知模型及 API。
- H5 的充值/退款按钮目前仍是前端本地加减，尚未接入真实资金流水。

### 第 4 步：业务模型补齐（已完成，等待审核）

#### 本阶段目标

把“账户有多少钱、积分为什么变化、退款由谁处理、通知是否已读、关键操作是谁做的”这些业务事实，变成数据库里可查询、可追溯的记录；同时先建立 Agent 任务状态表，为后面的人工作确认流程打基础。

#### 做了什么

1. 新增钱包账户和钱包流水。余额统一用“分”保存为整数，例如 200 元保存为 `20000`，避免小数计算出现精度误差。充值或退款不会只改余额，都会留下流水。
2. 新增退款申请。客户发起退款时余额不立即变化；员工或管理员审核通过后，才扣余额、写退款流水、写审计记录并发送站内通知。客户不能审核自己的退款。
3. 新增积分流水。每次积分变化都记录“变了多少、变化后剩多少、为什么变化”，客户可查询自己的积分明细。
4. 新增站内通知和已读状态。客户只能读取、标记自己的通知，不能操作别人的通知。
5. 新增审计日志。充值、退款审核、积分变动等关键动作会记录操作者、动作、业务对象和时间，方便排错和面试演示。
6. 新增 Agent 任务状态表。它保存发起人、任务类型、输入/输出、执行状态和“是否等待人工确认”。**当前它只是数据底座，没有接入 LangGraph，也没有调用大模型。**
7. 新增 Alembic 迁移 `0003_add_finance_and_notification_tables` 和 `0004_add_agent_task_states`。现有演示数据库已安全升级到最新版，没有删除预约、用户或钱包演示数据。
8. H5 和小程序的充值、退款、积分明细、站内通知页面改为真实后端接口，不再由前端本地伪造余额或通知。

#### 关键流程

```text
客户申请退款
  -> 后端创建“待审核”退款单，余额不动
  -> 员工/管理员审核通过
  -> 一次数据库提交：扣余额 + 写退款流水 + 写审计日志 + 发站内通知
  -> 客户刷新后看到新余额和通知
```

退款申请和扣钱分成两步：员工未同意前，客户的钱不会被提前扣掉；同意后，相关记录要么一起成功，要么一起失败，避免出现“钱扣了但没有记录”的半完成状态。

#### 和其他模块的关系

- 登录与权限：钱包、退款、通知都从 JWT 识别当前用户，不相信前端自己传来的用户 ID。
- H5 和小程序：两端复用真实 API，所以余额、积分和通知刷新后保持一致。
- 数据库迁移：通过 Alembic 新增表，不需要删库重建。
- 后续 Agent：任务状态表先保存 Agent 过程和人工确认状态；以后模型只能提建议，不能绕过确认直接改关键业务数据。

#### 验证结果

- `alembic upgrade head`：演示数据库成功升级到 `0004_add_agent_task_states`。
- `pytest tests/test_migrations.py -q`：`1 passed`，空 SQLite 数据库可从零升级到最新结构。
- `pytest tests/test_agent_task_state.py -q`：`1 passed`，Agent 状态可保存发起人、默认待处理状态和人工确认标记。
- `pytest -q`：`20 passed`，覆盖登录鉴权、预约归属、迁移、钱包充值、退款审核、通知已读和 Agent 状态模型。
- `python -m compileall -q backend migrations`、小程序 14 个 JavaScript 文件和 17 个 JSON 文件检查均通过。
- `git diff --check` 未发现空白符错误；仅有 Windows 换行符转换提示，不是代码错误。

仍有两类已知警告：第三方 TestClient 的弃用警告，以及 SQLite 删除存在循环外键的测试表时的警告。它们没有造成测试失败，也没有影响第 4 步功能；后续可以单独处理，不能说成已解决。

#### 面试讲法

> 钱包余额不是唯一事实，我为每次资金变化保留流水，并用整数分避免金额精度问题。退款采用“申请-审核-执行”三段式：申请不扣钱，审核通过后余额、流水、审计和通知在一次事务里完成。这样能防止越权，也能在问题发生时追溯是谁、何时、做了什么。AI 部分先保留任务状态和人工确认边界，模型未来只能辅助建议，关键写操作仍由人确认和后端事务完成。

#### 审核边界与下一步

第 4 步到这里结束，当前等待你的审核。没有开始第 5 步的客户业务 API 扩展、员工管理 Web、LangGraph、留存 Agent 或部署工作。

### 第 5 步：客户业务 API（已完成，等待审核）

#### 本阶段目标

让客户侧需要的业务数据都有明确的后端接口：预约、发型师和时间槽、会员资料、个人资料、充值、退款申请、积分流水和站内通知。后端从登录 token 识别客户，前端不再传一个手机号就代表身份。

#### 做了什么

1. 预约接口支持查询、创建和取消；取消时后端同时检查预约 ID 和当前用户 ID，客户不能取消别人的预约。
2. 会员接口支持当前客户创建/更新会员资料，以及按权限读取自己的会员信息；员工读取客户列表仍受员工角色保护。
3. 新增 `PATCH /api/profile`，客户可以保存姓名和 `MM-DD` 格式生日，手机号作为登录账号只读，避免页面修改了手机号但账号身份没有同步。
4. 钱包、退款、积分、通知接口沿用第 4 步的数据模型和事务规则，客户只能访问自己的数据，员工才可以审批退款。
5. 请求参数由 Pydantic 校验，金额必须大于 0，生日格式不正确会返回 422，未登录或越权会返回明确的 401/403/404。

#### 面试讲法

> 客户接口的核心不是“有几个 URL”，而是每个请求都从 JWT 还原当前用户，再用用户 ID 做查询条件。手机号、预约 ID 和会员 ID 都只是业务字段，不能用来替代身份校验。这样前端即使传错参数，也不能读写别人的数据。

### 第 6 步：H5 和小程序接真实接口（已完成，等待审核）

#### 本阶段目标

两个客户端都只负责展示页面状态，发型师、时间槽、预约、会员、钱包、退款、积分和通知全部来自同一套后端 API。

#### 做了什么

1. H5 继续使用统一 `apiRequest`，自动带 JWT，处理 401 登录失效和接口错误；小程序继续使用 `app.request`，统一带 token 并处理错误提示。
2. 两端的个人中心会从 `/api/auth/me`、`/api/members`、`/api/wallet`、`/api/points/transactions` 和 `/api/notifications` 刷新真实数据。
3. 两端个人资料页都能保存姓名和生日；手机号改为只读，避免产生“改了页面、没改账号”的假象。
4. 两端预约列表都有取消入口，调用真实 `DELETE /api/appointments/{id}`，取消后重新拉取列表；已取消或已完成预约不会重复显示取消按钮。
5. H5 和小程序到店记录页移除了写死的示例消费记录，改为展示后端返回的已完成预约；没有记录时显示空状态。

- 浏览器实际流程：注册/登录 -> 个人资料保存成功 -> 读取真实发型师和时间槽 -> 创建预约 -> 点击取消后显示“已取消”。
- H5 内嵌 JavaScript 语法检查通过；小程序 14 个 JavaScript 文件和 17 个 JSON 文件检查通过。
- `rg` 检查前端已无 `demo-*` ID 和写死的历史消费记录。

### 第 7 步：员工管理 Web（已完成）

#### 本阶段目标

2. 员工端登录仍然调用 `POST /api/auth/login`，token 只保存在浏览器的员工端存储键中；进入工作台后再次调用 `/api/auth/me`，确认当前账号角色是发型师或管理员。

#### 为什么这么做

客户端和员工端的使用目标不同。客户端关心自己的预约、余额和通知；员工端关心全店的预约、客户运营和待处理任务。如果把两者混在同一个页面，权限边界和演示流程都会变得不清楚，所以员工端使用独立入口和独立页面。

员工页面只负责组织数据和触发操作，真正的业务规则仍然在后端。例如员工点击“通过退款”时，后端才会在事务中扣余额、写流水、写审计和发通知。这样前端不能绕过后端直接伪造业务结果。

### 第 22 步：API 限流与统一错误响应（2026-07-13）

### 第 27 步：退出登录与用户友好错误提示（2026-07-13）

#### 本步改了什么

- 在 H5 “我的”页面的隐私政策入口下方增加独立的“退出登录”模块，隐私政策详情页不放退出按钮；按钮复用已有的 `logout()` 方法。
- 退出登录时删除浏览器保存的 JWT，清空当前用户状态，并返回登录主页。
- 增加统一错误提示映射：登录失败显示“账号或密码错误”，未登录或 Token 失效显示“登录已失效”，无权限显示“当前账号没有权限”，用户不存在显示“用户不存在”，重复注册显示“用户已存在，请直接登录”，参数错误和服务器错误也显示对应的中文提示。

#### 为什么改

用户直接看到 `401`、`403`、`409` 或 `500` 时，很难判断下一步该怎么做。把状态码保留给程序和日志，把提示翻译给用户，可以同时满足接口规范和实际使用体验。退出登录则让用户能够主动结束当前会话，方便更换账号和重新验证注册、登录流程。

#### 和其他模块的关系

- 隐私政策页和个人资料页共用现有 `logout()`，保证不同入口退出后的行为一致。
- JWT 仍由后端验证；前端删除 Token 只是结束本地会话，不能替代后端鉴权。
